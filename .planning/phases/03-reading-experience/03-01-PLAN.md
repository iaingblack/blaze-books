  <name>Task 1: Create ReadingMode enum and PageTextService with paragraph processing</name>
  <files>
    BlazeBooks/Models/ReadingMode.swift
    BlazeBooks/Services/PageTextService.swift
  </files>
  <action>
    **ReadingMode.swift:**
    Create a simple enum with two cases:
    ```swift
    enum ReadingMode: String, CaseIterable {
        case page = "Page"
        case rsvp = "RSVP"
    }
    ```
    No Identifiable or Codable needed -- just raw string for Picker display and CaseIterable.

    **PageTextService.swift:**
    Create a struct (not @Observable -- it's a stateless service like WordTokenizer) with:

    1. **ParagraphData model** (nested struct, Identifiable):
       - `id: String` — stable paragraph ID like "ch0-p3" (chapter index + paragraph index)
       - `text: String` — raw paragraph text
       - `wordRange: Range<Int>` — global word indices this paragraph covers (e.g., 45..<78)
       - `cachedTokens: [WordToken]` — pre-tokenized words for this paragraph (avoids re-tokenization on every word highlight change per Research Pitfall 1)

    2. **`splitIntoParagraphs(text:chapterIndex:)` method:**
       - Normalize line endings (\r\n -> \n)
       - Split on \n\n if present, else split on \n (same logic as existing ReadingView.loadChapter)
       - Trim whitespace, filter empty
       - For each paragraph: tokenize with WordTokenizer, compute wordRange from cumulative word count, cache tokens
       - Return [ParagraphData]

    3. **`paragraphIndex(forWordIndex:paragraphs:)` method:**
       - Given a global word index, return the index of the paragraph containing it
       - Use `paragraphs.firstIndex { $0.wordRange.contains(wordIndex) }` (simple linear scan is fine for 50-200 paragraphs)

    4. **`attributedString(for:highlightedWordIndex:)` method:**
       - Takes a ParagraphData and optional highlighted word index
       - Returns AttributedString of the paragraph text
       - If highlightedWordIndex is nil or not in this paragraph's wordRange, return plain AttributedString
       - If highlighted: compute localIndex = highlightedWordIndex - paragraph.wordRange.lowerBound
       - Look up the cached token at localIndex
       - Convert token.range (String.Index range) to AttributedString.Index range
       - Apply `.backgroundColor = .yellow.opacity(0.4)` and `.foregroundColor = .primary` to the highlighted word range
       - Do NOT animate the highlight (per Research anti-pattern: animation at TTS speed causes lag)
       - Use `.yellow.opacity(0.4)` for the highlight color (warm, readable, common in reading apps)

    **Important implementation details:**
    - Use the existing WordTokenizer for all tokenization (consistency with RSVPEngine/TTSService)
    - ParagraphData.cachedTokens uses the same NLLanguage.english pinning via WordTokenizer
    - The wordRange is a half-open Range<Int> where lowerBound is the first word's global index and upperBound is lowerBound + cachedTokens.count
  </action>

<task type="auto">
  <name>Task 2: Extend ReadingCoordinator with mode switching, highlighted word index, and debounced WPM</name>
  <files>
    BlazeBooks/Engines/ReadingCoordinator.swift
  </files>
  <action>
    Extend the existing ReadingCoordinator with three new capabilities:

    1. **Add readingMode property:**
       - `var readingMode: ReadingMode = .rsvp` (observable, defaults to RSVP since that's the existing behavior)

    2. **Add highlightedWordIndex computed property:**
       - Returns `isPlaying ? currentWordIndex : nil`
       - Page mode views observe this to show/hide word highlighting
       - When nil (paused), no word is highlighted in page mode (per Research recommendation: no "frozen highlight" when TTS is off)

    3. **Add switchMode(to:) method:**
       - Save currentWordIndex before switching
       - Track whether TTS was playing: `let wasTTSPlaying = isPlaying && isTTSEnabled`
       - Call stop() to halt both engines
       - Set readingMode = newMode
       - Restore currentWordIndex = savedIndex
       - Restore currentWord from rsvpEngine.word(at: savedIndex)
       - If wasTTSPlaying, call play() to resume in the new mode
       - This ensures READ-03: position is preserved across mode switches

    4. **Refactor setWPM for debounced TTS restart (NAV-01 + Research Pitfall 5):**
       - Split into two methods:
         a. `setWPM(_ wpm: Int)` — called continuously during slider drag. Updates currentWPM, applySpeedCap(), and updates rsvpEngine.setWPM(effectiveWPM) immediately. Does NOT restart TTS (avoids audio stuttering).
         b. `commitWPMChange()` — called when slider drag ends (onEditingChanged = false). If TTS is playing, stops and restarts from currentWordIndex with the new rate. This is the debounce point.
       - The existing setWPM method currently does both (update + TTS restart). Refactor it to only update engine/cap state. Move the TTS stop/restart logic to commitWPMChange().
       - commitWPMChange implementation:
         ```swift
         func commitWPMChange() {
             guard isTTSEnabled, isPlaying else { return }
             let resumeIndex = currentWordIndex
             ttsService.stop()
             if let voiceId = ttsService.currentVoiceIdentifier {
                 ttsService.setRate(speedCapService.wpmToRate(effectiveWPM, forVoice: voiceId))
             }
             ttsService.speak(fromWordIndex: resumeIndex)
         }
         ```

    5. **Update play() for page mode:**
       - In page mode with TTS enabled: TTS drives word advancement via onWordBoundary callbacks (same as RSVP mode)
       - In page mode with TTS disabled: No timer drives word advancement (user reads at their own pace; no RSVP timer needed)
       - Current play() already handles TTS mode correctly. Add a guard: if readingMode == .page && !isTTSEnabled, don't start the RSVPEngine timer (page mode without TTS is passive scrolling)

    **What NOT to change:**
    - Don't modify handleTTSWordBoundary -- it already updates currentWordIndex which page mode views will observe
    - Don't modify handleChapterComplete -- chapter auto-advance works the same in both modes
    - Don't change the onWordBoundary/onChapterComplete callback setup in init
  </action>
