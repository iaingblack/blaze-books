  <action>
**1. Define font size constants.** At the top of `ReadingView.swift` (outside the struct), add:
```swift
enum ReadingDefaults {
    static let fontSizeKey = "readingFontSize"
    static let defaultFontSize: Double = 17.0
    static let minFontSize: Double = 12.0
    static let maxFontSize: Double = 32.0
    static let fontSizeStep: Double = 2.0
}
```
This prevents Pitfall 5 (inconsistent @AppStorage defaults across views).

**2. Add `@AppStorage` to `ReadingView`.** Add property:
```swift
@AppStorage(ReadingDefaults.fontSizeKey) private var readingFontSize: Double = ReadingDefaults.defaultFontSize
```

**3. Create inline font size controls in ReadingView.** Add a private computed property `fontSizeControls` returning a view:
- `HStack(spacing: 16)` containing:
  - Decrease button: `Image(systemName: "textformat.size.smaller")` with `.font(.title3)`, action decreases by `fontSizeStep`, `.disabled(readingFontSize <= minFontSize)`, clamps to `max(minFontSize, readingFontSize - step)`
  - Size display: `Text("\(Int(readingFontSize))pt")` with `.font(.system(.caption, design: .rounded).monospacedDigit())`, fixed width ~45pt
  - Increase button: `Image(systemName: "textformat.size.larger")` with `.font(.title3)`, action increases by `fontSizeStep`, `.disabled(readingFontSize >= maxFontSize)`, clamps to `min(maxFontSize, readingFontSize + step)`
- Wrap in a small container: `.padding(.horizontal, 20).padding(.vertical, 8)`

**4. Place font size controls in ReadingView.** Add a `ToolbarItem(placement: .topBarTrailing)` containing the font size controls. This places the Aa controls in the top-right toolbar area (TOC button is top-left from Plan 01, mode toggle is principal/center).

**5. Update `PageModeView` to accept dynamic font size.** Add a property to `PageModeView`:
```swift
let readingFontSize: Double
```
Replace the hardcoded `.font(.system(size: 17))` on paragraph text (line 59) with:
```swift
.font(.system(size: readingFontSize))
```
Replace the hardcoded `.lineSpacing(7)` with proportionally scaled spacing:
```swift
.lineSpacing(readingFontSize * 0.41)
```
This maintains the ~41% ratio (7/17 = 0.41) established in the original design.

**6. Update all PageModeView call sites in ReadingView.** There are two `PageModeView(...)` initializations in ReadingView (broken chapter variant and TTS variant in `pageModeContentView`). Add `readingFontSize: readingFontSize` parameter to both.

**7. Update plain page mode content font size.** In `ReadingView.chapterContent` (the `plainPageModeContent` path), replace the hardcoded `.font(.system(size: 17))` on paragraph `Text` with `.font(.system(size: readingFontSize))`, and replace `.lineSpacing(7)` with `.lineSpacing(readingFontSize * 0.41)`.

**Important:** Do NOT apply font size to the RSVP display. RSVP uses a fixed 36pt monospaced font for ORP alignment -- changing it would break `characterWidth` calculations (per Research Open Question 3).
  </action>

  <action>
**1. Audit RSVPDisplayView dark mode behavior.** The RSVP view uses `Color.black` background and `.white` foreground text. Per Research recommendation and Pitfall 1, this is a **deliberate design choice** for RSVP reading comfort, not a dark mode bug. Do NOT change these colors -- RSVP stays dark in both light and dark system modes.

**2. Fix the idle state text color.** In `RSVPDisplayView.swift`, the idle state dash `"--"` uses `.foregroundStyle(.secondary.opacity(0.4))`. The `.secondary` semantic color on a `Color.black` background may render with poor contrast. Change this to `.foregroundStyle(.white.opacity(0.3))` for consistent contrast on the always-dark RSVP background.

**3. Add dark mode design documentation.** Add a doc comment to `RSVPDisplayView` explaining the deliberate dark-theme choice:
Update the existing struct-level doc comment to include a note like:
```
/// **Dark mode behavior:** The RSVP view intentionally uses a fixed dark background
/// (`Color.black`) regardless of system appearance. This is a deliberate design choice
/// for reading comfort during rapid word display, following the precedent set by Spritz
/// and similar RSVP readers. The app chrome (navigation bars, controls, chapter navigation)
/// follows the system dark/light mode setting automatically via SwiftUI semantic colors.
```

**4. Verify other views use semantic colors.** Quick audit of ReadingView.swift:
- `.foregroundStyle(.secondary)` -- semantic, adapts automatically. OK.
- `.foregroundStyle(.primary)` -- semantic. OK.
- `Color.accentColor` -- system accent. OK.
- `.background(.bar)` -- system material. OK.

No changes needed to ReadingView.swift or PageModeView.swift for dark mode -- they already use semantic colors throughout. APP-01 is satisfied by SwiftUI's default behavior plus the intentional RSVP dark theme.
